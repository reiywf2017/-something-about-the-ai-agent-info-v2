OpenCode 命令大全

CLI 命令 + 内置工具 + 内置代理 + 斜杠命令 | 数据来源：opencode.ai/docs + GitHub anomalyco/opencode

一、CLI 命令（终端执行）

用法：opencode <command> [options] | 完整文档：opencode.ai/docs/cli

命令 子命令 作用 & 使用场景 示例 一句话总结
核心执行（最常用）
opencode — 默认启动 TUI 交互界面。不带任何参数时直接打开终端 UI，这是最常用的方式。
支持 --continue（继续上次会话）、--model（指定模型）、--prompt（预设提示词） 		opencode
opencode -c # 继续上次会话
opencode -m anthropic/claude-sonnet-4-20250514
opencode --prompt "帮我review代码" 		日常开发就打这一个命令
run — 非交互模式执行。直接传入提示词，执行完自动退出。适合脚本、CI/CD、自动化场景。
支持 --format json（JSON输出）、--file（附加文件）、--attach（连接已有服务器） 		opencode run "解释这段代码"
opencode run --agent explore "找所有API路由"
opencode run --format json "列出所有TODO" > result.json
echo "修复这个bug" | opencode run # 管道输入 		CI/CD 和脚本自动化的首选
服务器模式（多人/远程协作）
serve — 启动无界面 HTTP 服务器。暴露完整的 OpenCode API，供远程客户端连接。后台常驻运行。
设置 OPENCODE_SERVER_PASSWORD 启用认证 		opencode serve
opencode serve --port 4096 --hostname 0.0.0.0
# 另一个终端连接：
opencode run --attach http://localhost:4096 "任务" 		后台跑 Agent 的方式，多客户端共享
web — 启动 Web 界面服务器。在浏览器中打开 OpenCode UI，适合不想用终端的场景。
支持 mDNS 局域网发现、CORS 配置 		opencode web
opencode web --port 4096
opencode web --mdns # 局域网其他设备可发现 		用浏览器操作 Agent
attach — 将终端 TUI 连接到远程服务器。配合 serve 或 web 使用。
支持 --dir（工作目录）、--session（指定会话） 		opencode attach http://10.20.30.40:4096
# 服务器 A 启动 serve，终端 B 用 attach 连上去 		远程连接已运行的 Agent
acp — 启动 ACP（Agent Client Protocol）服务器。通过 stdin/stdout 用 nd-JSON 通信，供 IDE 等客户端集成。
支持 --cwd、--port 		opencode acp
# IDE 扩展通过 stdin/stdout 与 Agent 通信 		给 IDE 扩展用的通信协议
代理管理
agent create 创建自定义代理。交互式引导，生成 Agent Markdown 文件（含 YAML frontmatter + 系统提示词）。
支持 --mode（primary/subagent/all）、--tools、--description 		opencode agent create # 交互式
opencode agent create --description "安全审查代理" --mode subagent --tools read,grep,glob 		让 LLM 帮你创建专用 Agent
list 列出所有可用代理。显示名称、模式、权限规则，内置代理排在前面。 opencode agent list 看看当前有哪些 Agent 可用
认证与模型
auth login 配置 API 密钥。基于 Models.dev 提供商列表，可为任何想用的提供商配置密钥。密钥存储在 ~/.local/share/opencode/auth.json。 opencode auth login # 交互式选择提供商 第一次用必须先登录
list 列出所有已认证的提供商。 opencode auth list 看看配了哪些模型提供商
logout 清除提供商凭据。 opencode auth logout 换个密钥时用
models — 列出所有可用模型。显示 provider/model 格式，可按提供商筛选。
--verbose 显示费用元数据；--refresh 刷新缓存 		opencode models
opencode models anthropic # 只看 Anthropic 的
opencode models --verbose --refresh 		查有哪些模型可以用
providers — 管理提供商认证。交互式向导或脚本化登录。支持 OAuth、API Key 等多种认证方式。 opencode providers # 交互式
opencode providers login anthropic --method api-key 		更高级的认证管理
MCP & 插件管理
mcp add 添加 MCP 服务器。交互式引导，支持本地和远程 MCP。 opencode mcp add 给 Agent 装新工具
list 列出 MCP 服务器及连接状态。状态：已连接 ✓、已禁用 ○、需认证 ⚠、失败 ✗ opencode mcp list 检查 MCP 连通性
auth 对支持 OAuth 的 MCP 服务器认证。 opencode mcp auth # 列出所有可认证的
opencode mcp auth my-server		 MCP OAuth 登录
logout 移除 MCP 服务器的 OAuth 凭据。 opencode mcp logout my-server MCP 登出
debug 调试 MCP 连接问题。排查为什么某个 MCP 连不上。 opencode mcp debug my-server MCP 排障专用
plugin — 安装 npm 插件。自动解析插件 manifest 并写入配置文件。别名：plug。
支持 --global（全局安装）、--force（强制重装） 		opencode plugin @scope/plugin-name
opencode plugin @scope/plugin-name --global 		像 npm install 一样装插件
会话管理
session list 列出所有会话。支持分页、JSON 输出。 opencode session list
opencode session list -n 10 --format json 		查看历史对话记录
delete 删除指定会话。 opencode session delete <sessionID> 清理不需要的会话
export — 导出会话为 JSON。敏感内容会自动替换为 [redacted]，可安全分享。 opencode export <sessionID> > session.json
opencode export # 交互式选择 		安全导出对话（自动脱敏）
import — 导入会话数据。从本地 JSON 或分享链接导入。 opencode import session.json
opencode import https://opncd.ai/s/abc123 		恢复或迁移会话
统计与运维
stats — Token 用量和费用统计。按天、按项目、按模型、按工具查看消耗。
--days、--tools、--models、--project 		opencode stats
opencode stats --days 7
opencode stats --tools 5 --models 		看看花了多少钱
db — 直接访问 SQLite 数据库。可运行 SQL 查询或打开交互式 shell。
子命令：path（打印路径）、migrate（迁移旧数据） 		opencode db # 交互式 shell
opencode db "SELECT id,title FROM session LIMIT 5" 		底层 DB 直接查
pr — 拉取 GitHub PR 并启动 OpenCode。自动 checkout PR 分支，处理 fork remote，检测 PR 中是否含分享的会话。 opencode pr 42 # checkout PR #42 并开始工作 Review PR 的最快方式
account login 登录 OpenCode 云服务。使用 device-code OAuth 模式，浏览器验证。 opencode account login https://app.opencode.ai 分享会话需要先登录
logout 登出 OpenCode 云服务。 opencode account logout —
switch 切换组织。 opencode account switch —
orgs 列出可用组织。 opencode account orgs —
github install 在仓库中安装 GitHub Agent。设置 GitHub Actions 工作流，实现 PR/Issue 自动化。 opencode github install 让 Agent 在 GitHub 上自动干活
run 运行 GitHub Agent。通常在 GitHub Actions 中调用。 opencode github run --token $GITHUB_TOKEN CI/CD 中调用
系统维护
upgrade — 升级 OpenCode。可指定目标版本和安装方式。 opencode upgrade # 升到最新
opencode upgrade v1.2.3 # 指定版本
opencode upgrade -m brew # 用 Homebrew 升级 		保持 OpenCode 最新
uninstall — 卸载 OpenCode。删除所有相关文件。可保留配置或数据。
--keep-config、--keep-data、--dry-run 		opencode uninstall
opencode uninstall --dry-run # 先看看会删什么 		彻底清理

二、内置工具（Agent 可调用的能力）

Agent 通过 ReAct 循环调用这些工具完成任务 | 通过 opencode.json 的 permission 字段控制权限（allow/deny/ask）

工具名 作用 使用场景 & 示例 一句话总结
bash 在项目环境中执行 shell 命令。可运行任何终端命令。 运行测试 npm test、查看状态 git status、安装依赖 pip install 等 Agent 的手——能执行命令
edit 通过精确字符串替换修改现有文件。Agent 修改代码的主要方式。 修复 bug：将 let x = 1 替换为 const x = 1 Agent 的笔——能改代码
write 创建新文件或覆盖现有文件。 创建新组件文件、生成配置文件、写测试文件 Agent 的笔——能建文件
read 读取文件内容。支持指定行范围，大文件可分段读取。 先读取文件再决定怎么改；查看配置内容 Agent 的眼——能看文件
grep 使用正则表达式搜索文件内容。底层使用 ripgrep，自动遵循 .gitignore。 搜索所有 TODO 注释、查找函数定义、定位错误信息 Agent 的搜索——能找内容
glob 通过模式匹配查找文件。返回按修改时间排序的路径。 找所有 TypeScript 文件 **/*.ts、找 src 下所有组件 Agent 的搜索——能找文件
lsp 与 LSP 服务器交互（实验性）。支持定义跳转、引用查找、悬停信息、调用层次等。
OpenCode 的独特优势——Agent 能像 IDE 一样理解代码语义		 跳转到函数定义、查找所有引用、查看类型信息 Agent 的 IDE——能理解代码语义
patch 对文件应用补丁。适用于各种来源的 diff。 应用 PR 的 diff、应用生成的补丁文件 批量修改的高级方式
skill 加载技能文件（SKILL.md）并注入到对话上下文中。 Agent 自动发现并加载项目级或全局级技能 让 Agent 瞬间学会新技能
todowrite 管理待办事项列表。创建和更新任务，跟踪多步骤操作进度。 复杂重构时创建任务清单，逐步完成打勾 Agent 的笔记本——能列计划
webfetch 获取网页内容。读取在线文档、API 文档等。 查阅第三方 API 文档、读取在线教程 Agent 能上网看文档
websearch 网络搜索。使用 Exa AI 搜索引擎，无需 API 密钥。
适合查超出训练数据截止日期的最新信息		 搜索最新的库版本、查找最新 API 变更 Agent 能上网搜索
question 向用户提问。包含标题、正文和选项列表，用户可选择或自由输入。 任务不明确时主动询问用户偏好；实现方案选择 Agent 不确定时会问人

三、内置代理类型

每个代理有不同的权限、工具集和行为模式 | 来源：GitHub 源码分析

代理名 模式 作用 & 权限特点 使用场景 一句话总结
build primary 默认主代理。权限最宽：允许提问、允许进入计划模式、可执行所有工具。
这是你在 TUI 中默认使用的代理 		日常开发：写代码、改 bug、重构、回答问题 万能主力，什么都能干
plan primary 计划模式代理。所有编辑工具被拒绝，只允许写 .opencode/plans/*.md。可退出计划模式。
只思考不动手，先规划再执行 		复杂任务拆解、架构设计、方案讨论 只动脑不动手的规划师
explore subagent 快速代码探索代理。默认拒绝所有工具，仅开放只读工具 + bash + web 工具。
速度快、权限小、安全 		快速浏览代码库、理解项目结构、查找特定代码 快速侦察兵，只看不改
general subagent 通用子代理。用于多步骤任务和并行工作。Todo 写入被禁用。
由主代理通过 task 工具调用 		并行执行子任务：一个修前端、一个修后端 主代理的助手，干杂活
compaction hidden 上下文压缩代理。自动总结长对话历史，保持在上下文窗口内。 系统自动调用，用户不直接使用 系统内部用，自动压缩对话
title hidden 标题生成代理。低温度（0.5），为会话生成简洁标题。 系统自动调用 系统内部用，自动起标题
summary hidden 摘要生成代理。为会话生成摘要。 系统自动调用 系统内部用，自动写摘要

五、Agent 协作机制详解：主代理 + 子代理 + 系统代理

用公司组织架构理解 Agent 体系 | 来源：opencode.ai/docs/agents

层级 角色 在 OpenCode 中对应什么 怎么交互 一句话理解
L0 你（老板） 终端用户。下达任务，查看结果。 在 TUI 中输入指令；按 Tab 切换主代理；输入 @代理名 调用子代理 你只跟主代理说话
L1
主代理		 Build
primary		 全栈工程师。默认主代理，所有工具全开：读写文件、执行命令、提问、进入计划模式。
打开 OpenCode 默认就是和 Build 对话 		· 你直接和它对话
· 按 Tab 可切换到 Plan
· 它有需要时会自动调用子代理（通过 task 工具） 		万能主力，什么都能干
Plan
primary		 架构师。规划专用，禁止改代码、禁止执行命令（需审批）。只能读代码 + 写计划文件到 .opencode/plans/*.md。
只思考不动手，先规划再执行 		· 按 Tab 切换到 Plan
· 适合"先听建议再决定"的场景 		只动脑不动手的规划师
L2
子代理		 General
subagent		 万能助手。完整工具访问（除 Todo 外），能改文件、能执行命令。适合多步骤任务和并行工作。
由主代理通过 task 工具自动调用，也可手动 @ 		· 主代理自动调用：Build 觉得任务需要帮助时自动派活
· 手动 @ 调用：@general 帮我改 backend/auth.ts 		主代理的助手，干杂活
Explore
subagent		 侦察兵。纯只读代理，不能改任何文件。只能读文件、搜索代码、上网查资料。速度快、权限小、安全。
最适合快速了解代码库结构 		· 主代理自动调用：Build 需要先了解代码库时
· 手动 @ 调用：@explore 找所有API路由 		快速侦察兵，只看不改
L3
系统代理
（隐藏）		 Compaction 上下文压缩。对话太长时自动将历史压缩为摘要，释放上下文窗口。 完全自动，你感知不到。可用 /compact 手动触发 后台自动压缩对话
Title 标题生成。每次会话开始后自动生成标题（低温度 0.5）。 完全自动 后台自动起标题
Summary 摘要生成。自动为会话生成摘要。 完全自动 后台自动写摘要

协作流程示例：你说"重构 auth 模块，把 JWT 换成 OAuth2"

你（老板）："重构 auth 模块"
  │
  ▼
Build（全栈工程师，主代理） 收到任务
  │
  ├─ 思考："任务很大，先了解现有代码"
  │  调用 task 工具 → @explore（侦察兵，子代理）
  │    │
  │    └─ Explore 只读扫描 auth 模块 → 返回文件列表和依赖关系
  │
  ├─ 收到 Explore 结果，制定重构计划
  │  调用 task 工具 → @general（万能助手，子代理）
  │    │
  │    └─ General 修改 backend/auth.ts（OAuth2 逻辑）
  │
  ├─ 同时：Build 自己修改 frontend 的登录组件
  │  （主代理 + 子代理 并行工作）
  │
  └─ 全部完成 → 回复你："重构完成，以下是改动..."
                （子代理结果汇报给 Build，Build 统一汇报给你）
关键要点：
① Build 是唯一和你直接对话的角色。子代理的结果汇报给 Build，Build 再汇报给你。
② 子代理执行时创建子会话，用快捷键 Leader+Right/Left 在主会话和子会话之间切换查看。
③ 子代理可以被 @ 手动调用，也可以被主代理自动调用。
④ Compaction/Title/Summary 在后台默默干活，你完全感知不到。
Build（全栈工程师） Plan（架构师） General（万能助手） Explore（侦察兵） 你还能自定义代理
类型 主代理 主代理 子代理 子代理 在 opencode.json 中定义：
"agent": {
  "reviewer": {
    "description": "代码审查",
    "mode": "subagent",
    "tools": {
      "write": false,
      "edit": false
    }
  }
}

或用 Markdown：
.opencode/agents/review.md

然后调用：
@reviewer 审查代码
谁在用 你直接对话（默认） 你直接对话（Tab切换） Build 调用 / 你 @ 调用 Build 调用 / 你 @ 调用
能改代码吗 能 不能（需审批） 能 不能（纯只读）
能执行命令吗 能 不能（需审批） 能 能（只读类）
什么时候用 日常开发全部场景 只想看方案，不想改代码 并行子任务 快速了解项目结构
类比 全栈工程师 架构师 万能助手 侦察兵

六、Agent Teams vs Subagent：两个不同的概念

Subagent 是内置的单兵模式 | Agent Teams 是插件提供的团队协作模式 | 来源：opencode-ensemble + PR #12711

对比维度 Subagent（子代理）
内置功能，开箱即用		 Agent Teams（代理团队）
需要安装插件 opencode-ensemble		 为什么这很重要
一句话定义 主代理的打工人——接到任务就干，干完就销毁，单向汇报 多个 Agent 组成团队——有分工、有协作、能并行、能互相通信 子代理 = 你一个人干不过来，雇几个临时工帮忙
团队模式 = 你组建一个正式项目组
安装方式 内置，无需任何配置
opencode 启动即可使用		 需要安装插件
opencode plugin install @hueyexe/opencode-ensemble
然后在 opencode.json 中配置 		Subagent 是 OpenCode 的基础能力
Agent Teams 是社区生态扩展
执行方式 串行 — 一个接一个执行
主代理 → 子代理A → 汇报 → 子代理B → 汇报 → ...		 并行 — 多个 Agent 同时工作
Agent A 和 Agent B 可以同时执行不同任务		 串行 = 你派一个人去查日志，回来再派另一个人去改代码
并行 = 两个人同时干不同的事
通信方式 单向汇报
子代理只能向主代理返回结果
子代理之间不能直接通信		 点对点通信
Agent 之间可以直接发消息
支持共享 Task Board（任务看板）		 单向 = 临时工只能跟老板汇报
点对点 = 团队成员之间可以直接沟通
生命周期 一次性（ disposable ）
任务完成即销毁，无状态保留		 持久化（ persistent ）
Agent 在整个任务周期内保持存活，记忆持续累积		 子代理用完即丢，不会记住上次干了什么
团队 Agent 有持续记忆，越干越懂上下文
模型混用 同模型
所有子代理使用当前会话设定的同一个模型		 多模型混用
不同 Agent 可使用不同模型（如 GPT-4o 做规划 + Claude 做编码）		 同模型 = 所有人用同一种工具
多模型 = 专家团队各有所长
文件隔离 共享目录
所有子代理在同一个工作目录中操作，可能冲突		 Git Worktree 隔离
每个 Agent 在独立的 worktree 分支中工作，互不干扰，完成后合并		 共享目录 = 多人同时改一个文件容易冲突
Worktree = 每个人有自己的副本，改完再合并
适合场景 · 单个复杂任务（重构一个模块）
· 需要分步执行的流水线任务
· 日常 90% 的开发工作 		· 大规模并行任务（同时修 10 个 bug）
· 多模型协作（规划 + 编码 + 审查分工）
· 需要多个 Agent 长时间协作的复杂项目 		日常开发用 Subagent 就够了
Agent Teams 是重型武器，适合大场面

实战对比：同一任务，两种方式

任务："重构认证模块，同时更新所有 API 测试"

执行步骤 Subagent 方式（串行） Agent Teams 方式（并行）
第1步 1 主代理（Build）分析任务
→ 派子代理 Explore 扫描认证模块代码结构 		1 同时启动两个 Agent：
· Agent A（Claude）：负责重构认证模块
· Agent B（GPT-4o）：负责更新 API 测试
第2步 2 Explore 汇报结构 → 主代理派 Build 开始重构 2 Agent A 和 B 同时工作，各自在独立 worktree 中操作
· Agent A 改认证代码 · Agent B 改测试代码
第3步 3 重构完成 → 主代理派 General 更新测试 3 Agent A 遇到问题 → 通过 Task Board 通知 Agent B 暂停
Agent B 调整测试策略后继续
第4步 4 测试更新完 → 主代理汇总结果 → 完成 4 两个 Agent 各自完成 → 自动合并 worktree → 完成
总耗时 4步串行 = 步骤之和 并行执行 ≈ 最长步骤的耗时

Agent Teams 安装与配置

步骤 命令 / 配置 说明
1. 安装插件 opencode plugin install @hueyexe/opencode-ensemble 安装 opencode-ensemble 插件，这是目前最成熟的 Agent Teams 实现
2. 配置 opencode.json 
{
  "plugins": {
    "@hueyexe/opencode-ensemble": {
      "teams": {
        "frontend": {
          "model": "anthropic/claude-sonnet-4",
          "worktree": true,
          "task": "专注于前端代码修改"
        },
        "backend": {
          "model": "openai/gpt-4o",
          "worktree": true,
          "task": "专注于后端 API 和数据库"
        }
      }
    }
  }
}
 · model：每个 Agent 可以用不同模型
· worktree：启用 Git Worktree 隔离
· task：定义 Agent 的职责范围
3. 启动团队 opencode run "重构认证模块" --team 使用 --team 参数启动团队协作模式
4. 官方支持 OpenCode 官方正在开发原生 Agent Teams 支持（PR #12711），目前尚未合并。
当前方案：使用 @hueyexe/opencode-ensemble 社区插件
未来方案：等官方 PR 合并后，Agent Teams 将成为 OpenCode 内置功能，无需插件
一句话总结：
Subagent = 主代理的临时工（串行、单向、一次性）—— 内置功能，日常够用
Agent Teams = 组建正式项目组（并行、互通、持久）—— 需装插件，重型任务才需要

你同事说的对：要用 Agent Teams 确实需要安装 @hueyexe/opencode-ensemble 插件。但 OpenCode 官方 PR #12711 正在推进原生支持，未来可能不再需要插件。
新手上路三步：
① opencode auth login — 配置模型提供商的 API 密钥
② opencode — 启动 TUI
③ /init — 让 Agent 自动理解你的项目

高频使用场景：
· 日常开发：opencode → 直接对话
· CI/CD 集成：opencode run "任务" --format json
· 代码审查：/review 或 opencode pr 42
· 费用查看：opencode stats --days 7
