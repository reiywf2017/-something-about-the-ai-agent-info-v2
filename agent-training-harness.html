Agent Harness：LLM 和 AI Agent 之间的关键中间层

理解 Harness = 理解为什么同一个模型在不同产品中表现天差地别 | 数据来源：parallel.ai, Anthropic, OpenAI, LangChain

AI Agent = LLM（大脑） + Harness（身体 + 工具 + 记忆 + 管控）

概念 所处层次 一句话定义 做什么（类比） 与 Harness 的关系 对开发者的意义
Harness
代理运行时		 整个中间层 包裹在 LLM 外面的完整软件基础设施，管理 LLM 自身之外的一切——工具调用、记忆、上下文、验证、循环 Agent 的身体
LLM 是大脑（只会生成文本），Harness 给它装上手（MCP 工具）、眼睛（观察结果）、记忆（上下文管理）、规矩（Hook 管控）。

没有 Harness，LLM 就是一个只会动嘴的专家		 Harness 本身
它是 MCP、Skill、Hook、Agent Loop、Memory 的容器和编排器。

Anthropic 官方称 Claude Agent SDK 为 "general-purpose agent harness"		 LLM 决定智商上限，Harness 决定能发挥多少。

同一个 Claude Sonnet 模型：
· 裸 API 调用 → 只能问答
· 套上 Claude Code 的 Harness → 能自主写代码、调工具、管项目
· 区别不在模型，在 Harness
Agent Loop
代理循环		 核心引擎 思考→行动→观察的迭代循环（ReAct 模式），驱动 Harness 不断运转直到任务完成 Agent 的心跳
每轮循环：LLM 思考下一步 → 执行工具 → 观察结果 → 再思考。

就像你排查 bug：猜原因→查日志→看报错→再猜→再查。Agent 干的跟你一样，只是快 100 倍		 Agent Loop 是 Harness 的核心执行引擎。

Harness 负责"有工具可用、有记忆可查"，Agent Loop 负责一轮一轮地用这些资源去完成任务		 理解了 Agent Loop，就理解了为什么 Agent 比 ChatGPT 贵——一个问题可能跑 10 轮 LLM 推理，不是一轮
MCP
Model Context Protocol		 连接层 Agent 连接外部系统的标准接口协议。让 Agent 能访问数据库、API、内部系统等 Agent 的手
LLM 只会动嘴（生成文本），MCP 让它能动手——能查日志、调 API、读数据库。

就像外卖小哥，脑子再好使，不给他电动车和手机也送不了外卖。MCP 就是电动车和手机		 MCP 是 Harness 的外部连接子系统。

Harness 通过 MCP 管理所有外部工具的注册、发现、调用、权限。MCP 是 Harness 给 LLM 装的"手"		 MCP 是开放标准——一个 MCP Server 开发一次，Claude Code / OpenCode / Copilot 都能用。你的投入不会白费
Skill
技能/工作流		 知识层 把重复操作固化为标准流程的指令包。教 Agent 一次，以后一键调用 同事的 SOP
你把排查 xMatter 的套路写成 SOP，以后谁拿到都能照着做。

Skill 就是这件事的 AI 版——你把解决问题的套路写成 Skill，Agent 以后无脑照着干		 Skill 是 Harness 的知识管理子系统。

Harness 负责在合适的时机发现、加载、执行 Skill。Skill 让 Harness 越用越聪明		 你教 Agent 一次，它记住一辈子。你的经验从"脑子里的直觉"变成"可分享的 Skill"——换项目换团队都能带走
Hook
钩子/拦截器		 管控层 Agent 执行动作前后的自动检查点。不符合规则直接拦住 安检门
MCP 让它能做，Skill 让它会做，Hook 决定它能不能做。想过关？先过检查		 Hook 是 Harness 的安全管控子系统。

Harness 在每次工具调用前后触发 Hook，实现自动 lint、测试、安全检查。Hook 让 Harness 可信		 你不用时刻盯着 Agent 怕它搞事。把规则写进 Hook，它会自动遵守——永不疲倦的 Code Reviewer
Memory
上下文管理		 记忆层 管理 Agent 的短期记忆（对话上下文）和长期记忆（跨会话持久化），防止上下文溢出 Agent 的笔记本
短期记忆 = 正在开会时记的笔记
长期记忆 = 项目文档库里归档的资料

上下文压缩（Compaction）= 笔记写不下时提炼摘要		 Memory 是 Harness 的上下文管理子系统。

Harness 在每轮 Agent Loop 中决定：给 LLM 看什么、隐藏什么、压缩什么。这是 Harness 最复杂的工作之一		 上下文管理决定了 Agent 能处理多复杂的任务。CLAUDE.md / AGENTS.md 就是注入到 Memory 的"项目背景知识"
Planning
任务规划		 规划层 将复杂任务拆解为子任务，制定执行顺序，逐步完成 项目经理
用户说"重构认证模块"，Planning 把它拆成：① 分析现有代码 → ② 设计新方案 → ③ 逐文件修改 → ④ 运行测试 → ⑤ 提交 PR		 Planning 是 Harness 的任务编排子系统。

OpenCode 的 plan 代理、Claude Code 的 Subagents 都是 Planning 的实现。Planning 让 Harness 能处理复杂任务		 没有 Planning 的 Agent 会"一口气干完"然后出错。有 Planning 的 Agent 会像靠谱的工程师一样分步骤来

同一个模型 + 不同的 Harness = 截然不同的产品

这就是为什么 Harness Engineering 正在成为 AI 领域的独立学科

场景 没有 Harness（裸 LLM API） 有 Harness（Claude Code / OpenCode / Copilot） 区别在哪
调用工具 LLM 只能生成文本建议"你可以运行 npm install"，但不会真的去运行 Harness 识别工具调用意图 → 通过 MCP 执行 → 结果反馈给 LLM → 继续推理 Harness 让 LLM 从"纸上谈兵"变成"亲自动手"
多轮任务 LLM 一次性生成全部代码，无法根据运行结果调整。错了就全错了 Agent Loop 让它写代码 → 运行 → 看报错 → 修改 → 再运行，循环迭代直到成功 Harness 让 LLM 从"一锤子买卖"变成"持续迭代"
长期项目 上下文窗口用完就忘记前面干了什么，下次开会从零开始 CLAUDE.md / AGENTS.md 提供项目记忆 + 自动 Compaction 压缩 + 跨会话持久化 Harness 让 LLM 从"金鱼记忆"变成"有笔记本的工程师"
安全保障 LLM 可能建议不安全的代码，没有自动检查机制 Hook 在提交前自动 lint → 运行测试 → 安全扫描，不通过就拦截 Harness 让 LLM 从"没人管的新人"变成"遵守规矩的老员工"
复用经验 每次都要从零教 LLM 你的团队规范，无法积累 Skill 固化最佳实践，新项目直接复用；MCP Server 开发一次三处可用 Harness 让 LLM 从"临时工"变成"越来越强的团队成员"
培训总结（一句话版）：

LLM 决定 Agent 的智商上限，Harness 决定 Agent 能发挥出多少智商。

Claude Code、OpenCode、GitHub Copilot 本质上是三个不同团队构建的三个不同的 Harness，它们可能使用相同的大模型（比如 Claude Sonnet），但因为 Harness 的实现不同，最终体验天差地别。

你之前学习的 MCP、Skill、Hook、Agent Loop——它们都是 Harness 的子组件。理解了 Harness 这个全局概念，你就能理解为什么这些子组件缺一不可。

话题二：Harness 到底是新概念还是换了个名字？

诚实地回答：它不是什么革命性新技术，而是一个"命名动作"——给一直存在的东西起了个新名字 | 但这个命名本身有工程意义

问题 事实 这意味着什么
时间线：AI 工程范式三次迁移（2022-2026）
2022-2024
Prompt Engineering		 核心问题："怎么写提示词？"
大家相信提示词的质量决定了 Agent 的成败。代表产物：ChatGPT、早期 Copilot（单文件补全）

死因：写得再好的提示词，如果 Agent 看不到项目中的工具函数，它还是会重复造轮子。这不是提示词问题，是信息问题。 		工程严谨性从未消失——它只是迁移到了更高的抽象层。

类比 就像邮件系统的演进：
· 2022：研究"怎么写一封好邮件"（Prompt）
· 2025：研究"收件箱里放什么"（Context）
· 2026：研究"设计整个邮件系统"（Harness）

每个新阶段包含前一阶段，不是替代。Harness 仍然需要好的 Prompt 和好的 Context。

关键人物：
· Mitchell Hashimoto（HashiCorp 创始人）2026.02 提出 Harness 概念
· OpenAI、Martin Fowler、Wharton 教授 Ethan Mollick 两周内同时独立发表相同结论
· 学术界称之为"多重发现"——说明大家的认知同时到达了同一个节点
2025 年中
Context Engineering		 核心问题："给什么信息？"
Shopify CEO Tobi Lütke 2025.06.19 一条推文引发行业词汇转换。Karpathy 随后定义："Context engineering is filling the context window with just the right information."

代表产物：Cursor（全代码库索引 + RAG）、CLAUDE.md / AGENTS.md

死因：上下文再完美，如果消费它的系统设计糟糕（循环逻辑差、错误恢复缺失、安全管控不足），照样失败。这不是信息问题，是系统问题。
2026 年 2 月
Harness Engineering		 核心问题："设计什么系统？"
Mitchell Hashimoto 总结："Every time the agent makes a mistake, change the system so that mistake structurally cannot recur." ——不要修提示词，改系统。

Martin Fowler 定义：Agent = Model + Harness
Anthropic 的 Claude Agent SDK 官方被称为 "general-purpose agent harness"

至今仅 2 个月。还在早期。
核心问题：Harness 到底是不是新东西？
它是什么？ Harness = Agent 的"非模型部分"

它包含的就是你已经知道的东西：
· Agent Loop（ReAct 循环）
· MCP（工具连接）
· Skills（知识复用）
· Hooks（安全管控）
· Memory（上下文管理）
· Planning（任务拆解）
· 权限系统、错误恢复、配置体系...

没有一个是新发明的。Harness 就是这些已有实践的总称。 		你的直觉是对的：Harness 不是革命性突破。

它不是一个新工具、新框架或新技术。它是一个认知框架——帮我们更好地讨论和思考已有的实践。

不要神化 媒体和自媒体确实在过度渲染。说"2025 Was Agents, 2026 Is Harnesses"更多是营销话术。

但有用 这个概念确实帮我们回答了一个关键问题："为什么同一个模型，在不同产品里表现天差地别？"——区别在 Harness。
那造这个词到底有什么用？——不是炒作的 3 个真实工程价值
价值 1
区分模型能力 vs 系统能力		 帮你回答一个实际问题：

"为什么同一个 Claude Sonnet 模型，在 Claude Code 里很强，在裸 API 里很弱？"——区别不在模型，在 Harness。

同理：Claude Code vs OpenCode vs Copilot，用着相同的模型，体验完全不同。差异 100% 来自 Harness。 		选 Agent 工具时，别只看模型——看 Harness 质量。

这就是为什么我们之前用 11 个维度对比三个工具——那 11 个维度本质上都在对比 Harness 的质量。
价值 2
有真实数据支撑		 三个团队的实证：

Manus 用相同模型，5 次重写 Harness，每次效果显著提升。模型没换，换了 5 次周围系统。

Vercel 砍掉 80% 的工具后，Agent 效果反而变好了。更少的工具 = 更少的歧义 = 更少的错误。

OpenAI Codex 团队 7 个工程师 5 个月写了 0 行手动代码，全靠设计 Harness 让 Agent 写了 100 万行代码。人类不再写代码，而是设计让 Agent 可靠地写代码的环境。 		Harness 的优化往往来自"减法"而非"加法"。

Vercel 的教训：给 Agent 太多工具反而让它困惑。好的 Harness 不是堆功能，而是只给必要的工具、只给必要的信息、只做必要的干预。
价值 3
定义了工程师的新角色		 工程师的角色正在变化：

旧角色 写代码 → 新角色 设计让 Agent 可靠写代码的环境

具体做什么：
① 把架构决策、编码规范写成 Agent 能读的文档（CLAUDE.md / AGENTS.md）
② 用自定义 linter 和测试机械地强制执行规则（而不是指望 Agent "自觉遵守"）
③ Agent 犯错时不修提示词——改系统让这个错不可能再发生 		每个 Agent 错误都是 Harness 的改进机会。

Hashimoto 的方法论：
Agent 犯错 → 不是模型的问题 → 是 Harness 没防住 → 改 Harness 让这类错误在结构上不可能再次发生

这就是"Harness Engineering"这个岗位的核心工作。
媒体炒作 vs 事实——培训时要澄清的误区
需要澄清的
❌ 媒体说法 ✅ 事实
"Harness 是 2026 年革命性新概念" 它是一个已有实践的新命名，才出现 2 个月
"Harness Engineering 是独立学科" 为时尚早，目前更多是认知框架
"Harness 让 Agent 突飞猛进" 它不创造新能力，而是让已有能力更可靠
"模型是商品，Harness 是护城河" 有一定道理但过于简化，模型质量仍然重要
"Harness 是一个工具/框架/产品" 它不是某个具体工具或产品。Claude Code 是一个产品，
它的 Harness 是产品内部的架构设计
培训建议：不要把 Harness 当成独立新概念讲，而是当"总结性框架"用。

① 先讲 MCP、Skill、Hook、Agent Loop 这些具体概念（你已经在做了）
② 最后用 Harness 收尾总结："我们今天学的这些东西加在一起，业界统称为 Harness。Agent = 模型 + Harness。"
③ 核心信息只有一个：决定 Agent 好不好用的，不仅仅是模型，更是 Harness 的质量
